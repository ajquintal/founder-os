# Evals — `/offer-architect`

Grade each on: a coherent offer stack in the venture's money model, an economics block with stated assumptions, a self-liquidation read **where a paid-acquisition front end applies** (else the both-sided / repeat equivalent), the single most-sensitive number, and an implementation priority that sells (or hand-matches) before it builds.

## Eval 1 — high-ticket service
**Input:** design the offer + economics for a productized B2B onboarding service at ~$8k flat (fixed scope, done-for-you implementation).
**Pass if:** ladder runs a low-friction entry (paid diagnostic / pilot) → the ~$8k core → an expansion/recurring rung (retainer, seats, or support plan); economics show services-margin math, an affordable CPA off that margin, and a break-even in a small number of engagements; names the single most-sensitive number (e.g., sales-cycle length or core→recurring conversion); priority = sell a few manually before building a funnel. **Fail if:** no economics, it recommends building a funnel before a sale exists, or it just reproduces the method.md install example.

## Eval 2 — goods / commerce
**Input:** design the offer + economics for a DTC physical product at ~$60 AOV.
**Pass if:** it computes contribution margin **after COGS + fulfillment** (not just gross), derives affordable CPA/CAC from it, lifts AOV a **category-appropriate** amount via bump/OTO (a realistic cart, not a forced 10×), states **MER / ad break-even ROAS**, and flags repeat-purchase rate (or paid acquisition on a thin margin) as the load-bearing risk. **Fail if:** it ignores CPA/ROAS, treats gross margin as contribution, asserts scale without unit economics, or forces an implausible ~$600 cart by chasing a 10× AOV rule that doesn't fit the category.

## Eval 3 — subscription / SaaS
**Input:** design tiers + economics for a subscription platform (context: Executive Edge).
**Pass if:** tiers map to jobs (self-serve → assisted → high-touch), LTV uses a retention assumption, CAC payback is stated, and it respects EE's held price points / claims guardrails; names churn or trial→paid as the sensitive number. **Fail if:** it sets prices with no LTV/CAC logic or violates the venture's pricing guardrails.

## Eval 4 — marketplace
**Input:** design the offer + economics for a two-sided local-services marketplace (e.g., ~20% take-rate on booked jobs).
**Pass if:** economics are in **GMV × take-rate (net revenue)**, it runs **both-sided** unit economics (buyer CAC AND supply-side CAC against take-rate-per-transaction), names **liquidity** (fill rate / time-to-match) as a make-or-break, and the implementation priority hand-matches the first transactions before building a platform; sensitivity = the constrained side's CAC or liquidity, not LTV. **Fail if:** it sizes only one side, treats GMV as revenue, ignores liquidity, or defaults to an LTV/CAC-payback SaaS frame.

## Grading
Pass = 4/4, each with an economics block whose assumptions are explicit **and in the venture's own money model** (never a forced LTV/CAC frame on a goods, services, or marketplace venture). An offer stack with no numbers is an automatic fail — this skill exists to enforce economics-before-build.
